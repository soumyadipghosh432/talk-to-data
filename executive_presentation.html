Talk-to-Data
Project Overview
01. EXECUTIVE BRIEFING

Conversational Business Intelligence with Secure Data Governance

Talk-to-Data is an enterprise-grade BI assistant that translates natural language questions into secure PostgreSQL queries. Built using LangGraph, SQLAlchemy, and modern LLMs (Gemini / AWS Bedrock), it bridges the gap between raw data storage and business executives while maintaining rigid row-level access control.

💡 The Problem Statement
Technical data extraction remains a bottleneck. Non-technical stakeholders depend on analysts for standard SQL summaries, slowing down decision loops. Standard LLM chatbots risk data leaking and lack localized row restrictions.
🚀 The Solution Value
Self-service BI. Direct conversational query processing with **automatic role-based query restriction**, ensuring users only query records within their assigned territory or department.
02. FEATURES & CAPABILITIES

System Features Built for Enterprise Readiness

Rather than a simple wrapper, the application implements enterprise capabilities required for production readiness and high-density telemetry reporting.

🛡️ Row-Level RBAC
Connects to role mappings to fetch active predicates (e.g. geographic_region = 'US'). Injects constraints directly into queries before compilation, preventing unauthorized lateral reads.
🗺️ Visual ER Schema
Includes an interactive, dynamic circular schema visualization tool in the Admin panel. Auto-renders database relationships, supporting mouse zoom and drag-to-pan actions.
📊 Operational Analytics
Full system health metrics compiling total queries, error logs, token breakdowns, latency charts, and a quick-export button to Excel-compatible CSVs.
03. DATA FLOW & ARCHITECTURE

The Stateful Pipeline Execution Path

Our workflow uses LangGraph, organizing SQL translation into a series of stateful checking nodes to enforce security and logical precision.

  • Security Inspection: Guardrails parse prompts to check for topics (Greeting vs. In-Scope vs. Abuse) contextually.
  • Automatic Policy Merging: Policy injector appends SQL clauses based on the mapped user profile rules.
  • Read-Only Enforcement & CTEs: Validator checks that queries start exclusively with SELECT or WITH, filtering out writing operations like UPDATE, INSERT, or ALTER.
04. DATA GOVERNANCE

Rigid Query Sanitization & Context Controls

Our engineering implementation focuses heavily on preventing prompt injections, model thinking leakage, and topic drift on ordinal operations.

🧹 Global Response Sanitizer
Modern reasoning models (like Gemini 2.5 and AWS Bedrock Claude) output raw XML thinking chains (e.g. <reasoning>...</reasoning>). We implement a global sanitizer utility that cleanses all LLM output, stripping reasoning text to protect SQL validation logic and UI bubbles.
📅 Context Preservation
Follow-up questions requesting offsets (e.g. "show me details for the 4th order") are prone to sorting drift. We mandate that follow-up queries inherit the parent's exact ORDER BY, WHERE, and table join configurations to preserve visual sequence offsets.

06. LLM PORTABILITY

Model Portability & Dynamic Swapping

The application is fully decoupled from a single model provider. Swapping LLMs is managed dynamically through a centralized configuration file.

⚙️ llm_config.yaml Profile
Switch the active provider instantly by modifying a single boolean toggle: 
models:
  - name: "GOOGLE_GEMINI_API"
    active: false
  - name: "AWS_BEDROCK_CLAUDE"
    active: true
