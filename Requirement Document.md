# 📑 Functional Requirement Document (FRD)

## 1. Introduction & Purpose

**Talk-to-Data** is an enterprise-grade AI chat application designed to democratize business intelligence. Traditional BI reporting can be rigid and bottlenecked by manual report creation. This application allows business users to ask conversational questions in plain English and automatically turns those questions into real-time database queries (`SELECT` statements) to fetch, summarize, and display data accurately and securely.

---

## 2. User Personas & Permissions

The system dynamically governs capabilities across two explicit user personas:

* 
**User:** Can register, log in, create new chats, search chat histories, ask conversational questions about data, provide feedback on responses, and export historical conversations. They are strictly bound to row-level data boundaries determined by their roles.


* 
**Admin:** Possesses all standard User capabilities. Additionally, holds exclusive access to an internal Admin Panel where they can view the corporate user directory, elevate users to Admin status, and assign or modify data access profiles and restriction rules.



---

## 3. Detailed Epic & Feature Specifications

### 🔑 Epic 1: Authentication & Session Onboarding

* 
**Split Homepage:** The default landing state splits the viewport to display side-by-side **Registration** and **Login** entry forms. The page architecture must be decoupled so that it can be completely bypassed in the future when Single Sign-On (SSO) integration occurs, directing authenticated network traffic straight to the main workspace layout.


* 
**Registration Rules:** Form validation restricts user creation strictly to usernames and passwords where the password must be between **5 and 20 characters**. On-screen text hints must continuously show these validation requirements to the user. New registrations defaults to a standard "User" security clearing.


* 
**Login Experience:** Users authenticating with correct credentials are instantly issued a secure session and redirected to the application's central Chat Interface.



### 💬 Epic 2: Core Chat Workspace & User Interface

* 
**Universal Header Layout:** Every page renders a fixed top header showcasing the company brand asset (`logo.png` for light backgrounds or `logo_dark.png` for dark layouts) on the upper left, adjacent to a customizable application title string defaulting to **"Talk-to-Data"**.


* 
**User Control Avatar:** The far right of the header contains a circular user profile icon displaying the initial character of the logged-in user's name. Clicking this icon drops down a persistent menu providing:


1. A global application Light/Dark interface theme switch.


2. A secure route link to the Admin Panel (hidden from standard Users).


3. A secure session Logout action.




* 
**Left Navigation Sidebar:** A distinct, scrollable left side panel houses the primary conversation context managers:


* 
*New Chat Trigger:* A button that immediately establishes a blank conversational session.


* 
*Real-time History Filter:* An interactive search field that filters historical logs instantly using keyword matching against user queries.


* 
*Historical Chat Registry:* A chronologically stacked list displaying past conversations. The display name of each item is derived dynamically from the first question asked in that session. If the stack overflows the display height, this navigation sidebar scrolls independently while keeping the main workspace stationary.




* 
**Main Chat Canvas:** Occupies the primary screen real estate. Content is categorized clearly into alternating chat bubbles:


* 
*User Bubbles:* Tagged with the individual's username.


* 
*AI Agent Bubbles:* Formally designated with an "AI" title header.




* 
**Awaiting Response States:** While backend tasks process query generation, the canvas renders a clean, non-flashy animated indicator showing that the AI engine is actively formulating a response.


* 
**Contextual Output Bubbles:** Every completion block generated by the AI includes a bottom-right action array providing:


* 
*Copy-to-Clipboard Icon:* Copies the bubble's raw text to the client system buffer and displays a temporary "Copied" confirmation tooltip.


* 
*Thumbs Up / Thumbs Down Interaction:* Collecting system sentiment. Clicking either option triggers an overlay modal allowing the user to type and submit optional feedback comments.


* 
*Latency Metric:* Positioned clearly on the outside boundary of the bubble, showing the exact time elapsed to process and return the response, formatted in decimal seconds (e.g., `1.2s` or `5.8s`).




* 
**Prompt Dock & Reporting Extractor:** Positioned permanently at the bottom center of the workspace. Features a broad conversational text entry lane alongside an export utility icon. Clicking this icon compiles the current chat history into a downloadable, production-grade PDF document.



### 🛡️ Epic 3: Enterprise Guardrails & Data Auditing

* 
**Input Scope Defense:** The conversational pipeline screens incoming text strings. Irrelevant, offensive, abusive, or out-of-scope inquiries are immediately rejected using polite, standardized system denial responses.


* 
**Data Masking & Row Filtering (RBAC):** The execution layer automatically intercepts queries to apply data isolation rules. Queries are restricted based on parameters mapped to the user's role (e.g., *only allowing data within the US geographic region*, *filtering out restricted product lines like A, B, and C*, or *isolating customers registered within a specific 30-day window*).

* 
**System Performance Analytics Dashboard (Admin Panel):** Exclusive to Admin accounts. Displays real-time operational metrics fetched from the system telemetry records. It includes a metrics card grid (Total Queries, Total Failures, Average Latency, Average Throughput in TPS, and Token breakdown) and a **Recent Query Execution Logs** data table displaying the last 10 execution logs with their exact generated SQL queries.

## 4. Database Schema Design (`init.sql`)

Below is the required relational database structure, engineered to support both core application operations and the granular, role-based metadata access rules:

```sql
-- Core User Account Tables
CREATE TABLE users (
    user_id SERIAL PRIMARY KEY,
    user_name VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NULL, -- Nullable to seamlessly accommodate future SSO migration
    access_type VARCHAR(50) DEFAULT 'User' NOT NULL, -- Core role: 'User' or 'Admin'
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Advanced Access Rules Engine (RBAC)
CREATE TABLE rules (
    rule_id SERIAL PRIMARY KEY,
    rule_name VARCHAR(100) UNIQUE NOT NULL,
    description TEXT,
    sql_predicate TEXT NOT NULL -- Contains the exact clause snippet, e.g., "region = 'US'" or "product_id NOT IN ('A','B','C')"
);

CREATE TABLE roles (
    role_id SERIAL PRIMARY KEY,
    role_name VARCHAR(100) UNIQUE NOT NULL,
    description TEXT
);

CREATE TABLE role_rules_mapping (
    role_id INTEGER REFERENCES roles(role_id) ON DELETE CASCADE,
    rule_id INTEGER REFERENCES rules(rule_id) ON DELETE CASCADE,
    PRIMARY KEY (role_id, rule_id)
);

CREATE TABLE user_roles_mapping (
    user_id INTEGER REFERENCES users(user_id) ON DELETE CASCADE,
    role_id INTEGER REFERENCES roles(role_id) ON DELETE CASCADE,
    PRIMARY KEY (user_id, role_id)
);

-- Chat Context Tracking (Structured JSONB Storage)
CREATE TABLE chat_history (
    chat_id UUID PRIMARY KEY,
    user_id INTEGER REFERENCES users(user_id) ON DELETE CASCADE,
    title VARCHAR(255) NOT NULL,
    history_data JSONB DEFAULT '[]'::jsonb NOT NULL, -- Array format: [{"role": "user", "content": "..."}, {"role": "ai", "content": "..."}]
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Quality Assurance Feedback Repository
CREATE TABLE feedback (
    feedback_id SERIAL PRIMARY KEY,
    chat_id UUID REFERENCES chat_history(chat_id) ON DELETE CASCADE,
    thumbs_up BOOLEAN DEFAULT FALSE NOT NULL,
    thumbs_down BOOLEAN DEFAULT FALSE NOT NULL,
    user_question TEXT NOT NULL,
    ai_response TEXT NOT NULL,
    comment TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Comprehensive Enterprise Telemetry Ledger
CREATE TABLE execution_log (
    log_id SERIAL PRIMARY KEY,
    chat_id UUID,
    user_id INTEGER,
    prompt_tokens INTEGER DEFAULT 0,
    completion_tokens INTEGER DEFAULT 0,
    total_tokens INTEGER DEFAULT 0,
    latency_ms INTEGER NOT NULL,
    throughput_tps NUMERIC(10, 2), -- Tokens per second
    llm_provider_utilized VARCHAR(100) NOT NULL,
    generated_sql_statement TEXT,
    execution_status VARCHAR(50) NOT NULL, -- 'SUCCESS', 'GUARDRAIL_DENIED', 'SQL_ERROR', etc.
    recorded_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

```

## 5. Multi-LLM Routing Strategy & Validation

The engine supports four runtime inference foundational models. Configuration parameters must be defined within a centralized structural environment file. One and only one provider entry is permitted to carry an active flag state at boot time. Additionally, the specific model name utilized by the active provider is parameterized inside the config file to allow easy switching.

```yaml
# llm_config.yaml
# Active Model Selection for Google Gemini:
# - "gemini-2.5-flash-lite"  (Recommended: Fast 1s response, healthy free-tier quota)
# - "gemini-3.1-flash-lite"  (Fast 2s response, healthy free-tier quota)
# - "gemini-3.5-flash"       (Newer preview, subject to temporary high demand limitations)
gemini_model: "gemini-2.5-flash-lite"

models:
  - name: "OPENAI_API"
    active: false
  - name: "GOOGLE_GEMINI_API"
    active: true
  - name: "AMAZON_NOVA_BEDROCK"
    active: false
  - name: "GPTOSS_20B_BEDROCK"
    active: false
```

### Server Execution Logic Loop

During startup, the FastAPI server reads the configuration file. If the verification step fails (e.g., zero models are active, or multiple models are set to active), the system aborts execution immediately with a structural initialization error.

```python
# System pseudo-constructor mapping
```

## 6. LangGraph State Machine Workflow & Security Guardrails

The processing pipeline operates as a formal LangGraph state machine. Every step acts as a secure, predictable processing node:

```
[Incoming User Prompt Context]
              │
              ▼
    [ Node 1: Guardrail Inspection ] ───(Violates Scope?)───► [Return Standard Denial Page]
              │
         (In Scope)
              ▼
    [ Node 2: RBAC Rule Injection ]
              │
              ▼
    [ Node 3: Structured Text-to-SQL Conversion ]
              │
              ▼
    [ Node 4: Strict Execution Validation ] ───(Failed SELECT Check/Risk Verified)───► [Flag Security Exception]
              │
         (Validated)
              ▼
    [ Node 5: Engine Query Execution ]
              │
              ▼
    [ Node 6: Comprehensive Synthesis ]
              │
              ▼
[ Append Structured Output Array to JSONB Storage ]

```

### Core Security & Processing Node Tasks

#### Node 1: Guardrail Inspection

An initial system prompt maps out boundaries regarding allowed topics. To resolve conversational context correctly, the **Conversation History** (up to 300 characters of recent turns) is loaded and parsed. Incoming statements are scanned to ensure they do not contain harmful intents, out-of-scope requests, or malicious syntax patterns. If an indicator breaks these rules, the graph stops execution immediately, logs a `GUARDRAIL_DENIED` status, and returns a secure, predefined denial message to the client.

#### Node 2: RBAC Rule Injection

The session fetches the active user's assigned access rules directly from the database schema. These extracted definitions are formatted as plain text constraint expressions and appended straight into the LLM context envelope as an un-bypassable system prompt requirement:

> 
> *"You are strictly required to filter all SQL query generation using the following operational rules: `WHERE geographic_region = 'US'` and `WHERE registration_date >= CURRENT_DATE - INTERVAL '30 days'`."* 
> 
> 

#### Node 3: Structured Text-to-SQL Conversion

The engine interprets the user's conversational prompt alongside the provided database schema mapping. To resolve vague references (like *"this"*, *"it"*, *"second one"*, *"details of the third one"*), the Conversation History is loaded and parsed. The prompt instructs the model to align ordinal selections back to the **primary listing entity** (e.g. Orders) rather than carrying over intermediate sub-query topics (e.g. Customers) unless explicitly asked. The schema configuration is maintained as a manually supplied document within the repository:

```json
{
  "database_tables": [
    {
      "table_name": "orders",
      "description": "Stores individual customer transactions, invoice balances, and tracking data.",
      "columns": [
        {"name": "order_id", "type": "INTEGER", "description": "Primary structural identity index key."},
        {"name": "geographic_region", "type": "VARCHAR", "description": "Global distribution territory value, e.g., 'US', 'EMEA', 'APAC'."},
        {"name": "total_amount", "type": "NUMERIC", "description": "Net transactional value calculated for the completed order processing."}
      ]
    }
  ]
}

```

#### Node 4: Strict Execution Validation

Before hitting the database connection pools, the generated SQL string passes through a strict validation engine:

1. 
*Structural Pattern Match:* The syntax is checked to verify it is explicitly and exclusively a **`SELECT`** command. If expressions like `DROP`, `INSERT`, `UPDATE`, `DELETE`, or structural execution variants like `GRANT` or `ALTER` are found, execution halts immediately with a security exception.


2. 
*SQL Injection Sanitization Check:* Characters or combinations that present an injection risk are filtered and blocked.



#### Node 5: Engine Query Execution

The clean, validated query string runs safely against the target PostgreSQL reader database instance.

#### Node 6: Comprehensive Synthesis

The raw rows and dataset results from the query are fed into the active LLM context. The model synthesizes these records into clear, human-readable text, structured markdown tables, and helpful summaries. The model is strictly instructed to:
1. **Never show the raw SQL statement** directly in the text response (which is displayed exclusively in the UI details drawer).
2. **Never prepends standard greeting prefixes** like *"Hello!"*, *"Hi!"*, or *"Hey!"*, delivering the analytical answer directly.
This complete output bundle is appended directly to the `history_data` JSONB table.

## 7. Operational Telemetry & OpenTelemetry Alignment

Backend processing operations monitor and capture precise execution data. Even if external LangSmith access tracking tokens are absent from the `.env` configuration file, the application captures all operational metrics locally. It transforms the gathered metrics into structured telemetry variables, writing them directly into the Postgres `execution_log` database table:

* 
**Prompt Token Ingestion Metrics:** Total token footprint calculated during prompt parsing.


* 
**Completion Token Export Metrics:** Total token footprint generated across model synthesis.


* 
**Latency Delta Calculation:** The precise round-trip travel duration (measured in milliseconds) from the moment the user's request hits the endpoint to the completion of the payload.


* **Operational Throughput Calculation:** Formulated explicitly via token consumption over time:

$$Throughput = \frac{Prompt\ Tokens + Completion\ Tokens}{Total\ Processing\ Latency\ (Seconds)}$$

## 8. Endpoint API Contract Specifications

### `POST /api/v1/auth/register`

* **Payload Ingestion:** `{"username": "testuser", "password": "securepassword123"}`
* 
**Processing Routine:** Validates password characters (minimum 5, maximum 20). Passwords are transformed using secure cryptographic hashing before database insert. The profile is created with a default `access_type` of `'User'`.


* **Response Payload:** `{"status": "success", "message": "User registry initialized."}`

### `POST /api/v1/auth/login`

* **Payload Ingestion:** `{"username": "testuser", "password": "securepassword123"}`
* 
**Processing Routine:** Validates parameters against the corresponding record row in the database. Generates a stateless JWT token string upon verification and assigns it to the client inside a secure, `httpOnly`, tracking cookie.


* **Response Payload:** `{"status": "authenticated", "access_type": "User"}`

### `POST /api/v1/chat/new`

* 
**Processing Routine:** Generates an isolated session UUID and creates a clean chat tracking row within the database schema.


* 
**Response Payload:** `{"chat_id": "4f92a3b8-12c4-4b81-8d2a-84bf7c83de12"}` 



### `POST /api/v1/chat/message`

* 
**Payload Ingestion:** `{"chat_id": "4f92a3b8-12c4-4b81-8d2a-84bf7c83de12", "message": "Show total order amounts inside the US region"}` 


* 
**Processing Routine:** Runs the multi-turn LangGraph agent processing loop, applies RBAC parameter controls, handles query validation checks, logs execution details, and tracks conversational data streams.


* 
**Response Payload:** Returns raw content data matching markdown layouts, dynamic tables, and processing time variables.



### `POST /api/v1/chat/feedback`

* 
**Payload Ingestion:** `{"chat_id": "4f92a3b8-12c4-4b81-8d2a-84bf7c83de12", "thumbs_up": true, "thumbs_down": false, "user_question": "...", "ai_response": "...", "comment": "Excellent data layout."}` 


* 
**Processing Routine:** Inserts feedback directly into the corresponding database mapping table.


* **Response Payload:** `{"status": "feedback_logged"}`

### `GET /api/v1/chat/export/{chat_id}`

* 
**Processing Routine:** Fetches historical payload array configurations bound to the unique chat identifier, compiles the conversational data stream, and handles document conversion.

**Response Payload:** Streams a clean `application/pdf` binary download directly to the client interface browser.

## 8. Admin Settings Panel Layout & Navigation

To provide a premium and clean user experience, the Admin Settings Panel is organized into four dedicated viewports accessible via sub-tabs:

1. **Analytics Dashboard**: Houses the system performance aggregates (KPI metric grids) and the recent query execution logs database table.
   * **Telemetry Pagination**: Telemetry logs are limited to the last 200 rows at the API layer, and paginated in pages of 20 records in the UI.
   * **Excel/CSV Export**: Includes an export button in the card header to trigger local downloads of the loaded 200 telemetry rows as a `.csv` file.
   * **Full SQL Extract Notice**: Features a label indicating that developers can query the complete execution logs beyond the 200 rows by running `SELECT * FROM execution_log ORDER BY recorded_at DESC;` directly on the database.
2. **RBAC Settings**: Houses the user directory table, role/rule mappings, and security configuration forms (role/rule creation, user-role associations).
   * **User Directory Pagination**: Paginates all loaded user profile records in pages of 20 records (unlimited database load).
   * **Role Configuration Search**: Features a search input field in the top right corner of the active role configurations card to filter lists by role name or description.
3. **Data Model View**: Houses the dynamic, interactive visual ER relationship diagram with zoom-in, zoom-out, reset, and drag-to-pan canvas operations.
4. **Model Settings**: Displays active LLM connection details and lists all available LLM integration pipelines (Gemini, OpenAI, Bedrock Claude, Amazon Nova, Bedrock GPT-OSS) and their active connection states.
